2. UEP v5.0 起動方法

※ 本番用起動スクリプトは 本番デプロイ\ フォルダにあります。

前提条件

  • Python 3.11以上
  • Node.js 18以上（フロントエンド用）
  • npm
  • backend/.env、frontend/.env.production の設定（本番デプロイ手順.md 参照）

2.1 一括起動

本番デプロイ\start-production-all.bat

バックエンドとフロントエンドを別ウィンドウで起動します。

2.2 個別起動

バックエンド（ターミナル1）:

本番デプロイ\start-backend-production.bat

フロントエンド（ターミナル2）:

本番デプロイ\start-frontend-production.bat
  • URL: フロント http://localhost:3000、バックエンド http://localhost:8080
  • API Docs: 本番モード時は無効
  • 停止: 各ウィンドウで Ctrl+C

※ デモユーザー（kaho0525 / 0525、developer / dev123）でログイン可能です。

3. 統合セキュリティ・防衛プラットフォーム 起動方法

UEP v5.0とは別の個別システムです。UEP起動時には含まれません。

前提条件

  • UEP の backend/venv が作成済みであること（初回は UEP の start-backend.bat を1回実行して venv を作成）
  • Node.js 18以上（フロントエンド用）

3.1 バックエンドの起動

cd C:\uep-v5-ultimate-enterprise-platform\security_defense_platform
.\start-backend.bat
  • URL: http://localhost:9001
  • API Docs: http://localhost:9001/docs
  • 停止: Ctrl+C

※ ポート9001が使用中の場合は、9002〜9010の空きポートで自動起動します。

3.2 フロントエンドの起動

PowerShell:

cd C:\uep-v5-ultimate-enterprise-platform\security_defense_platform
.\start-frontend.ps1

コマンドプロンプト:

cd C:\uep-v5-ultimate-enterprise-platform\security_defense_platform
.\start-frontend.bat

直接 npm で起動:

cd C:\uep-v5-ultimate-enterprise-platform\security_defense_platform\frontend
npm install   # 初回のみ
npm start
  • URL: http://localhost:3001
  • 停止: Ctrl+C

※ デモユーザー（kaho0525 / 0525、developer / dev123）でログイン可能です。

起動順序

  1. バックエンド（9001）を先に起動
  2. フロントエンド（3001）を起動

7.6 要対応サマリ

画面上部に全ドメインの要対応・要確認件数を一覧表示します。各チップをクリックすると、該当タブに遷移します。

  • 製造: 要メンテナンス設備 + 高重大度の異常
  • 医療: 要確認のAI診断所見 + 高重大度の異常
  • 金融: 高リスクスコア + 要確認の取引監視
  • セキュリティ: Critical/High のセキュリティイベント
  • インフラ: 未対応の監視アラート + Pendingデプロイ
  • DX: 失敗・待機中のパイプライン
  • コンプライアンス: 未対応の監査指摘 + 要対応の規制
  • サプライチェーン: 要発注の在庫

最終更新時刻も表示されます。

7.10.1 スタンドアロン起動（推奨）

  • フロントエンド: http://localhost:3002

7.10.2 機能

機能 内容
Docker クリーンアップ 未使用コンテナ・イメージ・ボリューム・ビルドキャッシュの削除
Cドライブ スキャン node_modules, venv, __pycache__, ビルド成果物等の検出

WSL VHDX Docker WSL2 の VHDX サイズ表示（diskpart compact で圧縮可能）

※ 並列スキャンで高速化。本番起動は 7.10.1.5 を参照。

7.10.3 確保可能な容量の目安

対象 目安 備考
Docker イメージ・コンテナ 数GB〜20GB以上 docker system prune -a で未使用分を削除（ボリュームは残す）
node_modules 各数百MB〜 削除後 npm install で再生成
venv, __pycache__ 数百MB〜 削除後、起動・実行で再生成
build/, dist/, .next/ 数百MB〜 ビルドで再生成
postgres_data, minio_data 等 数GB〜 開発用のみ削除可。本番データがある場合は削除しないこと
Windows.old 10〜30GB ディスクのクリーンアップ → 以前の Windows のインストール
WSL VHDX 10〜50GB以上 wsl --shutdown 後、diskpart で compact

メニューから選択: 1.検出のみ / 2.Docker クリーンアップ / 3.Cドライブ+一時フォルダ削除 / 4.全て実行

10. トラブルシューティング

ポートが使用できない（WinError 10013 / 10048）

  • 8000: Windows予約範囲のため使用不可。UEPバックエンドは8080を使用
  • 9001使用中: 統合セキュリティ・防衛プラットフォームは9002〜9010で自動的に空きポートを探して起動

フロントエンドが起動しない（PowerShellでバッチ実行時）

  • .\start-frontend.ps1 を使用（PowerShell用）
  • または cd frontend のうえで npm start を直接実行

認証エラー（Not authenticated）

  • ログイン画面でデモアカウントでログイン
  • バックエンドが起動しているか確認（8080 または 9001）

データが表示されない

  • バックエンドが起動しているか確認
  • ブラウザの開発者ツール（F12）でネットワークエラーを確認
  • CORSエラーの場合、バックエンドのCORS設定を確認
