デモの進め方

デモは面談の流れに合わせて実施する。事前に環境を起動し、手順書を手元に用意しておく。

  • 実績・プロジェクトを聞かれたとき 「実際に動作しているシステムがあります。画面共有でデモンストレーションできます」と伝え、了承されたら画面共有でデモを行う。
  • 技術的な深掘りをされたとき UEP v5.0 を起動し、Swagger UI とフロントのダッシュボードを中心に説明する。
  • 時間が足りない／話題が別のとき 無理にデモはせず、口頭で概要（Level 1～5 統合、技術スタック、品質指標など）を説明する。
  • 「デモを見せてください」と言われたとき すぐに画面共有に切り替え、「面談当日の起動手順」以降に従ってデモを行う。
  • 事前にしておくこと start-all.bat で起動できる状態にしておく。本手順書を開いておく。Zoom の画面共有を確認しておく。

よくある質問

Q1: なぜWindows環境？ デモ環境としてWindowsデスクトップPCを想定。WSLやDocker Desktopが不要で簡単に起動できるため。

Q2: 認証情報の変更は？ backend/auth/routes.py の _init_demo_users で変更可能。現在は kaho0525 / kaho052514。

Q3: 本番環境での使用は？ 開発・デモ用。本番ではセキュリティ・DB・環境変数・ログの見直しが必要。

Q4: パフォーマンスは？ 開発環境向け。本番ではDB最適化、キャッシング、ロードバランシング、CDNを推奨。

Q5: レベル5グローバルエンタープライズ対応とは？ グローバル展開を想定した設計。マルチリージョン、高可用性、コンプライアンス、災害復旧まで考慮。Level 1（7コア）から Level 5（グローバル）まで段階的に拡張できる構成。

「グローバル展開を想定した設計で、マルチリージョン、高可用性、コンプライアンス、災害復旧まで考慮しています。Level 1の7コアから、Level 5のグローバルエンタープライズまで段階的に拡張できる構成です。」

トラブル時の対応

バックエンドが起動しない: python --version 確認 → venv 再作成 → それでもダメなら Swagger UI のスクリーンショットで説明

フロントエンドが起動しない: node --version 確認 → node_modules 削除 → npm install → それでもダメなら Swagger UI でAPI実演

ログインできない: バックエンド起動確認 → 認証情報確認 → それでもダメなら Swagger UI でAPI実演

画面共有ができない: スクリーンショット共有 または コードを直接共有して説明

強調すべきポイント

  1. 実際に動作するシステムであること（個人開発の実績を証明）
  2. エンタープライズレベルの品質（稼働率99.995%、推論レイテンシ<10ms）
  3. 技術的な深さ（MLOps/LLMOps、生成AIの実装経験）
  4. 統合プラットフォームとしての完成度（Level 1～5 グローバルエンタープライズ対応、2025-2026最新4技術統合、ビジネス領域12種：DX・EX・CX・UX・BX・HX・SX・BPR・BPM・CRM・ERP・HRTech）
