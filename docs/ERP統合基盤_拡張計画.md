# ERP・統合基盤システム 拡張計画

**作成日**: 2026年3月10日  
**目的**: 企業案件で多い統合基盤システム（ERP、レガシー刷新、データ連携、DX推進）の追加構築

---

## 結論：追加構築は可能です

既存のUEP v5.0基盤（Docker、Kubernetes、Terraform、Kafka、MinIO、API Gateway、認証・認可）を活用し、以下のシステムを段階的に追加できます。

---

## 1. ERP（統合基幹業務システム）

**会計・人事・販売・購買などを統合した基幹システムの導入・刷新**

### 既存の土台

| 領域 | 既存システム | 状態 |
|------|-------------|------|
| 会計 | 個人会計（UEP版・backend/personal_accounting）、FinTech、金融・経済統合 | ✅ あり |
| 人事 | 統合ビジネスプラットフォーム（障害者雇用、オンボーディング、スキルマッチング） | ✅ あり |
| 販売 | 小売モジュール、顧客対応・CX（チケット、チャットボット） | △ 部分的 |
| 購買 | industry_leader（調達）、製造業モジュール | △ 部分的 |

### 追加で必要なモジュール

| モジュール | 内容 | 工数目安 |
|------------|------|----------|
| 販売管理 | 受注・出荷・請求・売上集計、在庫連携 | 40～80h |
| 購買管理 | 発注・入荷・支払・仕入集計、発注点管理 | 40～80h |
| 会計統合 | 個人会計を法人向けに拡張、勘定科目・仕訳・決算 | 60～120h |
| ERP統合API | 会計・人事・販売・購買の横断データ連携 | 20～40h |

### 実装方針

- `backend/erp/` を新規作成
- 既存の `unified_business_platform`、`personal_accounting`、`fintech` とAPI連携
- Kafkaでイベント駆動（受注→出荷→請求→会計 の流れ）

---

## 2. レガシー刷新・クラウド移行

**メインフレームやオンプレからクラウドERPへの移行**

### 既存の土台

| 要素 | 既存 | 状態 |
|------|------|------|
| コンテナ化 | Docker、Kubernetes | ✅ あり |
| IaC | Terraform、Helm | ✅ あり |
| CI/CD | GitHub Actions | ✅ あり |
| マイクロサービス | Kong、Envoy、API Gateway | ✅ あり |
| 監視 | Prometheus、Grafana | ✅ あり |

### 追加で必要なもの

| 項目 | 内容 | 工数目安 |
|------|------|----------|
| データ移行ツール | CSV/Excel/DBからの一括インポート、マッピング | 20～40h |
| レガシーAPIアダプター | 既存システムとの連携用プロキシ | 20～40h |
| 移行検証ダッシュボード | 移行前後のデータ比較、整合性チェック | 20～40h |

### 実装方針

- `backend/infra_builder` の拡張、または `tools/legacy-migration/` を新規作成
- 既存のデータレイク（MinIO）、イベントストリーミング（Kafka）を活用

## 3. データ連携基盤

**基幹システムと周辺システムの連携・データ統合**

### 既存の土台

| 要素 | 既存 | 状態 |
|------|------|------|
| イベントストリーミング | Apache Kafka | ✅ あり |
| オブジェクトストレージ | MinIO（データレイク） | ✅ あり |
| API Gateway | Kong、Envoy | ✅ あり |
| 認証・認可 | JWT、OAuth2/OIDC | ✅ あり |

### 追加で必要なもの

| 項目 | 内容 | 工数目安 |
|------|------|----------|
| データ連携オーケストレーター | 基幹↔周辺のデータ同期ルール、スケジュール | 40～80h |
| ETL/ELTパイプライン | データ変換・ロード、dbt または Python | 40～80h |
| 連携監視ダッシュボード | 連携状況、エラー、レイテンシ | 20～40h |

### 実装方針

- Kafka Connect または 自前のPython ETL を `backend/data_lake` に追加
- 既存の Prometheus/Grafana で監視拡張

---

## 5. 実装優先順位（推奨）

| 順位 | システム | 理由 | 工数目安 |
|------|----------|------|----------|
| 1 | データ連携基盤 | 既存インフラが最も活きる。他システムの土台 | 80～160h |
| 2 | ERP（販売・購買） | 会計・人事は既存あり。販売・購買でERPを完成 | 100～200h |
| 3 | レガシー移行ツール | 案件で需要が高い | 60～120h |
| 4 | DX推進基盤拡張 | 既存の業務効率化を強化 | 40～80h |

---

## 6. 面談・案件でのアピール

**「追加構築可能です」の根拠**

- 既に100システム以上の実装実績
- 統合基盤（Kafka、MinIO、API Gateway、認証）は稼働済み
- 会計・人事・業務効率化のモジュールはある
- 販売・購買・データ連携の拡張は、既存アーキテクチャに沿って追加可能

**面談での一言例**

> 「ERP・統合基盤の基盤は既に実装済みです。会計・人事・業務効率化のモジュールがあり、販売・購買・データ連携基盤の追加構築も可能です。レガシー刷新・クラウド移行の経験も、Docker、Kubernetes、Terraformでの実装で実証しています。」
