# 大規模運用システム 設計案

医療系・航空系・宇宙系スタンドアロンを、大規模運用可能なシステムへ拡張するための設計案。

---

## 1. 現状（質素）と目標（大規模）の比較

| 観点 | 現状（質素） | 大規模運用目標 |
|------|-------------|----------------|
| アーキテクチャ | 単一FastAPI、メモリ内データ | マイクロサービス、永続化DB |
| スケーラビリティ | 垂直のみ | 水平スケール、オートスケール |
| 可用性 | 単一障害点 | 冗長化、マルチAZ |
| 監視 | なし | Prometheus/Grafana、APM、アラート |
| データ | ハードコード | PostgreSQL/Redis、バックアップ |
| 認証・認可 | なし | OAuth2/OIDC、RBAC |
| CI/CD | 手動 | パイプライン、Blue-Green |
| コンプライアンス | なし | 監査ログ、暗号化、アクセス制御 |

## 3. 各ドメイン別 大規模化設計

### 3.1 医療系（Medical）

| 項目 | 設計内容 |
|------|----------|
| **データ層** | PostgreSQL（患者・診断・バイタル）、Redis（セッション・キャッシュ） |
| **FHIR準拠** | FHIR R4 リソース型、検索パラメータ、バリデーション |
| **AI診断** | 非同期ジョブキュー（Celery/RQ）、モデルバージョン管理 |
| **コンプライアンス** | 監査ログ（誰が・いつ・何を）、暗号化（保存時・転送時）、アクセス制御 |
| **スケール** | 患者数・病院数に応じたDBシャーディング検討 |

### 3.2 航空系（Aviation）

| 項目 | 設計内容 |
|------|----------|
| **データ層** | PostgreSQL（フライト・空港・遅延履歴）、TimescaleDB（時系列分析） |
| **リアルタイム** | WebSocket / Server-Sent Events で運航状況配信 |
| **外部連携** | 空港API、気象API、遅延情報API のアダプタ層 |
| **スケール** | 1日数千フライト想定、時系列データのパーティション |

### 3.3 宇宙系（Space）

| 項目 | 設計内容 |
|------|----------|
| **データ層** | PostgreSQL（衛星・打ち上げ）、Object Storage（画像・軌道データ） |
| **外部API** | NASA API、SpaceX API 等のプロキシ・キャッシュ層 |
| **大量データ** | 衛星画像のストリーミング、メタデータのみDB保存 |
| **スケール** | 衛星数・打ち上げ頻度に応じたキャッシュ戦略 |

## 4. インフラ・運用

### 4.1 コンテナ・オーケストレーション

```yaml
# Kubernetes デプロイ例（医療系）
apiVersion: apps/v1
kind: Deployment
metadata:
  name: medical-api
spec:
  replicas: 3
  selector:
    matchLabels:
      app: medical-api
  template:
    spec:
      containers:
      - name: medical-api
        image: registry/medical-api:latest
        resources:
          requests:
            memory: "256Mi"
            cpu: "100m"
          limits:
            memory: "512Mi"
            cpu: "500m"
        livenessProbe:
          httpGet:
            path: /health
            port: 8000
        readinessProbe:
          httpGet:
            path: /ready
            port: 8000
---
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: medical-api-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: medical-api
  minReplicas: 2
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
```

### 4.2 監視・アラート

| ツール | 用途 |
|--------|------|
| **Prometheus** | メトリクス収集（リクエスト数、レイテンシ、エラー率） |
| **Grafana** | ダッシュボード、アラート通知 |
| **Loki** | ログ集約・検索 |
| **Jaeger / OpenTelemetry** | 分散トレーシング |
| **PagerDuty / Slack** | インシデント通知 |

### 4.3 CI/CD

```
Git Push → GitHub Actions / GitLab CI
  → テスト（pytest, lint）
  → イメージビルド（Docker）
  → レジストリプッシュ
  → K8s デプロイ（Argo CD / Helm）
  → スモークテスト
```

### 4.4 認証・認可

- **認証**: Keycloak / Auth0 / Cognito（OAuth2/OIDC）
- **認可**: RBAC（ロールベース）、APIキー管理
- **API Gateway**: レート制限、認証チェック、ログ

## 5. 段階的導入ロードマップ

| フェーズ | 内容 | 工数目安 |
|---------|------|----------|
| **Phase 1** | DB導入（PostgreSQL）、ヘルスチェック、Docker化 | 1〜2週間 |
| **Phase 2** | Redis、認証（JWT/OAuth2）、監査ログ | 2〜3週間 |
| **Phase 3** | Kubernetes デプロイ、HPA、監視（Prometheus/Grafana） | 2〜3週間 |
| **Phase 4** | CI/CD、Blue-Green、DR構成 | 2〜4週間 |
| **Phase 5** | 分散トレーシング、高度なアラート、コンプライアンス強化 | 継続 |
