### 2. 仕様書やチケットの内容が曖昧で、解釈に幅があると感じた時、あなたは通常どのように対応しますか？ *

**選択肢**:
- PMや担当者にチャットで質問し、回答が来るまで待つ
- 不明点をリストアップし、MTGを設定して確認する
- **自分なりの仮説（松竹梅の案）を立てて、「これで進めて良いか」を提案ベースで確認する** ← 推奨
- とりあえず実装してみて、PRレビューで確認してもらう

**推奨理由**: 提案ベースで確認すると、相手の負荷を減らしつつ、自分の理解を示せる。個人開発で「課題→工夫→成果」の流れを実践してきた経験と整合する。

### 3. 実装中、チャットのテキストだけでは説明しづらい複雑な問題に直面しました。最初に取る行動として最も近いものはどれですか？ *

**選択肢**:
- テキストで説明するのは誤解を招くため、すぐにPMや有識者に「今から通話できますか？」とメンションを送る
- **状況・試したこと・推測される原因をドキュメントやスクリーンショットにまとめ、まずはテキストで共有して指示を仰ぐ** ← 推奨
- 次の定例ミーティングまで待ち、そこで画面共有しながら説明する

**推奨理由**: 非同期コミュニケーションを重視。3年9ヶ月のリモート勤務で、ドキュメント・スクショでの共有を実践してきた。

### 4. 完全リモート環境において、PMやチームメンバーに「今、何が進んでいるか」を可視化するために、あなたは日次でどのようなアウトプット（Gitのプッシュ頻度やSlack報告のタイミングなど）を心がけていますか？ *

**回答案**:
```
個人開発では、Gitのコミット・プッシュを日次で実施し、進捗を可視化してきました。チーム開発では、以下を心がけます。
・Git: 作業単位でコミット・プッシュ。PRは小さい単位で作成
・Slack等: タスク開始・完了・ブロッカー発生時に報告。定例前に進捗サマリを共有
・タスク管理ツール: ステータスを随時更新し、誰が見ても「今どこか」が分かる状態を維持
```

### 5. タスクが想定より難航し、今日中の完了が難しそうだと気づいた時、あなたは通常どう動きますか？ *

**回答案**:
```
早期にPM・担当者へ報告します。具体的には、
・現状・残タスク・見込み完了時刻を簡潔に共有
・「〇日までに完了可能」「〇の部分は明日に回す」など、代替案を提示
・ブロッカーがあれば併せて伝え、支援を依頼
50名規模のチーム統括経験から、報告の遅れが全体に影響することを学んでいます。
```

### 6. 開発における生成AI（Claude/Cursor/Codex等）でコードを書く際、品質担保のために「必ず行っていること」を教えてください。 *

**回答案**:
```
①生成コードの理解: AIの提案をそのまま使わず、ロジック・エッジケースを自分で確認
②テスト: 単体テスト（Pytest）で動作確認。カバレッジ80%以上を目安
③レビュー: セキュリティ（pip-audit）、スタイル（flake8/black/mypy）でチェック
④判断は人間主導: 要件定義・設計・最終判断は自分で行い、AIは補助として活用
```

### 7. あなたが「業務委託エンジニア」として働く上で心掛けていることを教えてください *

**回答案**:
```
・納期遵守: 約束した期日を守る。難航しそうな場合は早期報告
・品質: 稼働率99.995%、推論レイテンシ10ms未満を個人開発で達成した水準を維持
・コミュニケーション: 非同期を前提に、ドキュメント・スクショで状況を共有。相手の負荷を考慮
・スコープの明確化: 曖昧な部分は提案ベースで確認し、認識齟齬を防ぐ
```

### 10. チーム開発において、他部署や非エンジニア（PMや営業等）と意見が対立した際、相手の意図を汲み取るために、あなたが最も意識していることは何ですか？ *

**回答案**:
```
・「なぜ」を聞く: 表面的な対立ではなく、背景の目的・制約を理解する
・技術用語を避ける: 相手が理解できる言葉で、ビジネス価値・影響の観点から説明
・選択肢を提示: 「A案だと〇〇、B案だと〇〇」と比較し、相手が判断しやすくする
50名規模のチーム統括で、非エンジニアとの調整経験を活かします。
```
