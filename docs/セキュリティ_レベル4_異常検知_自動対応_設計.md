## 1. 概要

### 1.1 目的

- **異常検知**: ルールベースに加え、機械学習による未知の脅威の検知
- **自動対応（SOAR）**: アラート発生時の自動封鎖・通知・プレイブック実行


## 2. 異常検知の設計

### 2.1 検知方式の比較

| 方式 | 検知対象 | メリット | デメリット |
|------|----------|----------|------------|
| **ルールベース** | 既知の攻撃パターン | 高精度、低誤検知 | 未知の攻撃に弱い |
| **統計的異常検知** | ベースラインからの逸脱 | 未知の攻撃に強い | チューニングが必要 |
| **機械学習** | 複雑なパターン | 学習により精度向上 | 学習データ・計算リソース必要 |

### 2.2 実装方式

#### 方式A: 統計的異常検知（軽量）

**対象メトリクス例:**
- ログイン失敗回数（時間あたり）
- 同一IPからのリクエスト数
- 異常な時間帯のアクセス

**アルゴリズム:**
- Z-Score: 平均から標準偏差の N 倍以上離れた値を異常と判定
- IQR: 四分位範囲外を異常と判定

```python
# 疑似コード
def is_anomaly(value: float, baseline_mean: float, baseline_std: float, threshold: float = 3.0) -> bool:
    if baseline_std == 0:
        return False
    z = abs(value - baseline_mean) / baseline_std
    return z > threshold
```

#### 方式B: 機械学習（scikit-learn 等）

**対象:**
- ネットワークトラフィックの特徴量（パケット数、バイト数、送信先ポート数）
- 認証ログの特徴量（失敗回数、時間帯、ユーザー）

**モデル:**
- Isolation Forest: 外れ値検出
- One-Class SVM: 正常データのみで学習
- Autoencoder: 再構成誤差が大きいものを異常と判定

```python
# 疑似コード
from sklearn.ensemble import IsolationForest

model = IsolationForest(contamination=0.01, random_state=42)
model.fit(normal_features)
predictions = model.predict(new_features)  # -1 = 異常
```

### 2.3 データフロー

```
1. ログを OpenSearch から取得（または Kafka からストリーミング）
2. 特徴量抽出（時間窓、集計、正規化）
3. ベースライン計算（過去 N 日分の平均・標準偏差）
4. 異常スコア判定
5. スコアが閾値超なら SecurityEvent として登録
6. UEP Security Center に表示
```

## 3. 自動対応（SOAR）の設計

### 3.1 自動対応の種類

| 種類 | 内容 | 例 |
|------|------|-----|
| **通知** | アラート発生時のメール・Slack 通知 | 重大アラート時に Slack へ投稿 |
| **封鎖** | IP の自動ブロック | ブルートフォース検知時に Firewall でブロック |
| **プレイブック** | 定義された手順の自動実行 | インシデントチケット自動作成 |
| **エスカレーション** | 人間による判断が必要な場合の通知 | Critical 時に管理者へ連絡 |

### 3.2 プレイブック定義（例）

```yaml
# プレイブック例: ブルートフォース検知時
playbook_id: brute_force_response
trigger:
  - event_type: ids_alert
    rule_id: "2000001"  # SSH Scan
  - event_type: auth_failure
    threshold: 10  # 10回/5分
actions:
  - type: notify
    slack_channel: "#security-alerts"
    message: "ブルートフォース検知: {{ source_ip }}"
  - type: block_ip
    target: "{{ source_ip }}"
    duration: 3600  # 1時間
  - type: create_incident
    title: "ブルートフォース攻撃: {{ source_ip }}"
    severity: high
```

### 3.3 自動対応エンジンの設計

```
[アラート発生]
      │
      ▼
[プレイブックマッチング]
  - トリガー条件を満たすか
  - 対象プレイブックを特定
      │
      ▼
[アクション実行キュー]
  - 非同期で実行（Celery / バックグラウンドタスク）
  - 失敗時はリトライ・エスカレーション
      │
      ▼
[実行結果]
  - 成功/失敗をログ
  - UEP インシデントに記録
```

### 3.4 実装コンポーネント

| コンポーネント | 役割 |
|----------------|------|
| **PlaybookEngine** | プレイブックの読み込み・マッチング |
| **ActionExecutor** | 実行（通知、封鎖、API 呼び出し） |
| **BlockIPService** | ファイアウォール・iptables の API 連携 |
| **NotifyService** | Slack / メール / Webhook 送信 |

## 4. セキュリティ考慮事項

### 4.1 自動封鎖のリスク

- **誤検知**: 正常な業務 IP をブロックする可能性
- **対策**: ホワイトリスト、段階的封鎖（最初は警告のみ）、人間による承認フロー

### 4.2 プレイブックの改ざん防止

- プレイブックファイルの整合性チェック（ハッシュ）
- 変更履歴の管理（Git）
- 本番反映前のレビュー承認

### 4.3 権限分離

- 自動対応を実行するサービスアカウントは最小権限
- 封鎖・解除は別の承認フローを経由

| 4 | 機械学習モデル（Isolation Forest） | 中 |
